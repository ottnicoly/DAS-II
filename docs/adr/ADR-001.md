# ADR-001: Uso de Azure Functions no formato serverless para ingestão de dados

**Status:** Proposto  |  **Data:** 2026-04-05  |  **Autores:** Nicoly Cristina Ott

## Contexto

A VendaMais Distribuidora Ltda. é uma empresa de médio porte com operações em quatro estados, 18 representantes comerciais e volume de aproximadamente 3.500 pedidos/mês, operando um ERP proprietário desde 2019.

O problema central é a ausência de visibilidade consolidada sobre indicadores operacionais: relatórios levam até 2 dias úteis para serem compilados manualmente, a inadimplência é calculada em planilhas desatualizadas e a diretoria toma decisões com dados de até 30 dias de defasagem.

A decisão se restringe à **camada de ingestão** — ponto de entrada dos dados do ERP antes de qualquer processamento ou persistência. Os requisitos principais são: escalabilidade elástica, baixo custo em períodos ociosos e rastreabilidade de execuções.

## Decisão

Adotamos **Azure Functions no formato serverless** como estratégia de ingestão de dados da plataforma.

Cada fonte de dados será mapeada para uma ou mais funções com responsabilidade única, acionadas por triggers nativos do Azure (HTTP, Service Bus, Event Grid ou Timer). A infraestrutura será gerenciada pelo Azure, sem provisionamento de servidores.

Funções críticas com requisito de baixa latência utilizarão o **plano Premium** para eliminar cold start. Funções de baixa frequência permanecerão no **plano Consumption**. Fluxos de ingestão com duração superior a 10 minutos serão implementados com **Durable Functions**.

## Consequências

(+) Custo proporcional ao uso real: sem cobrança em períodos ociosos, o que é determinante na fase atual de crescimento da plataforma.

(+) Escalabilidade automática de zero a N instâncias sem intervenção manual, absorvendo picos de tráfego sem superprovisionamento.

(+) Deploy independente por função, permitindo atualizar integrações de fontes distintas sem risco de regressão entre elas.

(+) Integração nativa com Azure Service Bus e Event Grid, eliminando código de roteamento ou polling nas funções.

(+) Rastreabilidade out-of-the-box via Application Insights integrado ao runtime do Azure Functions.

(-) Vendor lock-in: migração para outra cloud exigirá reescrita das funções e substituição dos triggers.

(-) Cold start no plano Consumption pode introduzir latência de alguns segundos nas primeiras execuções após inatividade, impactando integrações síncronas.

(-) Observabilidade distribuída é mais complexa: correlacionar traces de uma ingestão de ponta a ponta entre múltiplas funções requer instrumentação cuidadosa.

(-) Testabilidade local limitada: o emulador local (Azure Functions Core Tools) não replica fielmente todos os comportamentos de produção, exigindo testes de integração em ambiente cloud.

## Alternativas rejeitadas

- **Microsserviços em contêineres (AKS):** custo fixo de nós do cluster mesmo em períodos ociosos e overhead operacional elevado para uma equipe pequena, sem benefício proporcional no contexto de ingestão event-driven.

- **Azure Logic Apps:** flexibilidade insuficiente para lógicas de validação e transformação complexas; versionamento via configuração visual dificulta revisão de código e rastreabilidade em repositório Git.

- **Azure Stream Analytics + Event Hub:** custo fixo e complexidade operacional inadequados para o perfil atual de volume; solução de streaming contínuo é overkill para fontes que entregam dados em batch ou por polling.
