
# ADR-002: Definição da Estratégia de Armazenamento para a Camada Analítica

Status: Proposto  |  **Data:** 2026-04-06  |  **Autores:** Matheus Campos

## Contexto

A VendaMais Distribuidora Ltda. necessita centralizar dados provenientes de seu ERP proprietário para eliminar a dependência de planilhas Excel e garantir uma "única fonte da verdade". O volume atual de dados é de aproximadamente 3.500 pedidos/mês e uma carteira de 1.200 clientes, o que representa um volume de dados médio, mas com alta necessidade de integridade referencial (especialmente nos módulos Financeiro e Estoque).

Os requisitos principais incluem:

Consistência: Garantir que os cálculos de faturamento e inadimplência sejam idênticos em todos os dashboards.

Integração: Facilidade de consumo nativo pelo Power BI.

Manutenibilidade: A equipe interna deve ser capaz de operar a solução sem uma curva de aprendizado excessiva em tecnologias de Big Data.

Custo-benefício: A solução deve ser escalável, mas financeiramente viável para uma empresa de médio porte.

## Decisão

Decidimos adotar o Azure SQL Database (nível de serviço General Purpose ou Serverless) como o repositório central de dados (Data Warehouse/Data Marts) para a plataforma VendaMais.

Esta escolha baseia-se na maturidade do motor SQL Server para modelagem dimensional (Star Schema/Snowflake) e na sua capacidade de lidar com as transformações via Azure Functions (Python) de forma eficiente através de queries relacionais e procedures, se necessário.

## Consequências

(+) Sinergia com Power BI: O Azure SQL possui conectores nativos otimizados (DirectQuery ou Import) que facilitam a atualização automática diária e o gerenciamento de segurança.

(+) Integridade de Dados: O suporte total a transações ACID garante que, durante o processo de transformação (Camada 4), não existam estados parciais de dados que possam corromper os indicadores financeiros.

(+) Curva de Aprendizado Baixa: Sendo uma tecnologia baseada em SQL, a manutenção por parte da equipe interna de TI da VendaMais é simplificada, facilitando a sustentação do projeto.

(+) Segurança: Integração nativa com Azure Active Directory (Microsoft Entra ID) e recursos como criptografia em repouso e mascaramento de dados sensíveis.

(-) Escalabilidade Horizontal Limitada: Diferente de soluções de Big Data (como Spark), o escalonamento é primariamente vertical, embora o volume projetado para a VendaMais esteja muito abaixo dos limites de performance do serviço.

(-) Custo de Armazenamento: Comparado ao Blob Storage bruto, o custo por GB no Azure SQL é superior, o que exige uma política de limpeza de dados históricos ou arquivamento de logs antigos.

## Alternativas rejeitadas

- **Azure Synapse Analytics (Dedicated SQL Pool):** Azure Synapse Analytics (Dedicated SQL Pool): Rejeitado por ser uma solução voltada para volumes de dados na escala de Terabytes (Big Data). Para a realidade da VendaMais, o custo seria proibitivo e a complexidade arquitetural desnecessária.

- **Azure Cosmos DB (NoSQL):** Rejeitado devido à natureza altamente relacional dos dados de ERP. A necessidade de realizar joins complexos entre pedidos, clientes e estoque tornaria a modelagem documental ineficiente e cara para fins analíticos.

- **Azure Blob Storage com Azure Synapse Serverless (Data Lakehouse):** Embora moderno, foi rejeitado para evitar a complexidade de gerenciar arquivos Parquet e esquemas externos (External Tables) neste momento inicial, priorizando a estabilidade e o conhecimento prévio da equipe em bancos relacionais.
